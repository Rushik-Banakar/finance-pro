1. PROJECT OVERVIEW
==================================================
- Project Name: Finance Pro (Money Manager)
- Problem Statement: Modern personal finance tracking is often tedious, lacks proactive risk warnings, and offers static dashboards that fail to guide strategic financial decision-making. Standard tools do not provide contextualized, data-anchored advice or predictive indicators regarding overspending and saving milestones.
- Business Objective: Create a secure, intelligent, and comprehensive personal ledger and wealth advisor that combines structured double-entry ledger tracking with data science indicators, machine-learning analytics, and generative AI personas. The application helps users track account balances, manage budgets, prevent overspending, identify saving opportunities, and receive real-time, data-grounded wealth coaching.
- Target Users: Tech-savvy professionals, freelancers, retail investors, and business owners looking for a modern ledger platform with secure locks, automated budget calculations, and built-in financial intelligence.
- Key Features:
  * Multi-Account Ledger: Track liquid and credit balances across Savings, Current, CreditCard, Cash, and Wallet accounts.
  * Transaction Lifecycle: Log incomes, expenses, and account transfers with automatic balance reconciliation.
  * Smart Budget Control: Define outlay thresholds for expense categories with real-time risk evaluation (Critical, Warning, Healthy).
  * Category Splits: Split existing categories into two distinct groupings with automatic transaction migration options (move all, move manual).
  * Machine-Learning Analytics: Expense forecasting (LinearRegression), spending outlier flags (IsolationForest), and daily spending behavioral grouping (KMeans Clustering).
  * AI Coach Panel: A conversational financial advisor offering multiple specialized personas (CFO, Health Diagnostics, Savings Optimizer, Budget Control, Affordability Evaluation, Master Advisor).
  * PIN Lock Security: Security PIN authorization for private dashboards, custom lock durations, session timeouts, and hidden balance masking.
- System Architecture Overview:
  * Frontend: Single Page Application (SPA) built using React (v19), TypeScript, and Vite. State management uses Zustand. Server state sync and caching are managed via TanStack React Query. UI components are built using Tailwind CSS, Framer Motion for animations, and Recharts for interactive visualization.
  * Backend: FastAPI (Python ASGI framework) implementing RESTful API routers. Session tracking is managed using JWT Bearer tokens. Relational storage is driven by SQLite via SQLAlchemy ORM.
  * ML & AI pipeline: Scikit-Learn handles statistical predictions. A failover service coordinates connections to Groq, OpenAI, Google Gemini, and Anthropic APIs, defaulting to local rule-based templates if offline.
- High-Level Workflow:
  1. A user authenticates and gains access to their ledger. If enabled, the application triggers a PinLock screen.
  2. The user inputs financial events. The backend adjusts the respective account balances atomically.
  3. The Analytics component requests data science KPIs and ML outputs. The backend aggregates transactions using Pandas, fits ML models, and reports forecasts, anomalies, and KMeans groups.
  4. The user interacts with the AI Advisor. The router builds a JSON payload summarizing current balances, budgets, trends, and scores.
  5. The LLM Service reviews the context, selects the appropriate prompt agent, calls the best available LLM provider, sanitizes the response to strip markdown formatting in favor of raw HTML, and renders the advice in the Coach panel.

2. TECHNOLOGIES USED
==================================================
* React (v19.2.6)
  - What it is: A declarative UI library for building component-based SPA web interfaces.
  - Why used: Enables a modular, component-driven architecture with fast virtual DOM reconciliation, crucial for real-time dashboard updates.
  - Where used: Entire frontend interface (`frontend/src/App.tsx`, `frontend/src/pages/*`, `frontend/src/components/*`).
  - Interview Explanation: "React 19 was selected to drive a high-fidelity web client. By utilizing a virtual DOM, React allows the application to render real-time dashboard card updates, financial charts, and conversational panels without full-page reloads."

* TypeScript (~v6.0.2)
  - What it is: A typed superset of JavaScript that compiles to plain JavaScript.
  - Why used: Provides compile-time type verification, preventing runtime errors in complex financial data flows.
  - Where used: All frontend code files.
  - Interview Explanation: "TypeScript was implemented across the frontend to enforce type safety on API schemas, State models, and React Component props. This guarantees that parameters like transaction quantities, categories, and account balances adhere strictly to predefined data structures before reaching the UI."

* Vite (v8.0.12)
  - What it is: A modern frontend build tool and dev server.
  - Why used: Offers instant server start-up and fast Hot Module Replacement (HMR) to accelerate development iterations.
  - Where used: Frontend builder tool config (`frontend/vite.config.ts`, `frontend/package.json`).
  - Interview Explanation: "Vite replaced legacy bundlers to significantly reduce build times. Its Hot Module Replacement allows for real-time styling updates and quick state reflections, optimizing front-end iteration efficiency."

* TanStack React Query (v5.100.14)
  - What it is: An asynchronous state management library for React.
  - Why used: Manages remote server-state caching, loading states, and automatic cache invalidation during ledger mutations.
  - Where used: Data fetching hooks in frontend pages (`frontend/src/pages/Analytics.tsx`, `frontend/src/components/CoachWidget.tsx`).
  - Interview Explanation: "React Query is used to manage remote server state. It decouples local UI state from database records, automatically caching dashboard queries and executing invalidations on mutations so that charts refresh instantly when a transaction is logged."

* Zustand (v5.0.13)
  - What it is: A small, fast, and scalable state-management library.
  - Why used: Provides lightweight, boilerplate-free global state for user sessions and client-side UI configurations.
  - Where used: Global stores (`frontend/src/store/useAuthStore.ts`, `frontend/src/store/useUiStore.ts`).
  - Interview Explanation: "Zustand was chosen for client-side state management, specifically for the user session and visual settings. Its lack of context boilerplate makes it extremely lightweight compared to Redux, allowing direct, hook-based access to state properties like `isLocked` or `hideBalance` across nested components."

* Tailwind CSS (v4.3.0)
  - What it is: A utility-first CSS framework.
  - Why used: Accelerates rapid responsive design with flexible classes, custom theme tokens, and modern color palettes.
  - Where used: All HTML/TSX elements.
  - Interview Explanation: "Tailwind CSS v4 is used to construct a modern, premium dark-themed interface. Its utility classes enable responsive layouts, micro-animations, and glassmorphism elements, maintaining visual consistency across mobile and desktop screens."

* Framer Motion (v12.40.0)
  - What it is: An animation library for React.
  - Why used: Drives high-performance, smooth transitions and gestures for the slide-in coach widgets and popup modals.
  - Where used: Page transitions and UI overlays.
  - Interview Explanation: "Framer Motion is integrated to implement smooth UI micro-animations. It controls the slide-in and fade effects of the AI Coach widget and PIN lock screens, improving user engagement through responsive visual feedback."

* Recharts (v3.8.1)
  - What it is: A composable charting library built on React components.
  - Why used: Renders interactive, responsive SVG charts.
  - Where used: Category outlay and distribution diagrams (`frontend/src/pages/Analytics.tsx`).
  - Interview Explanation: "Recharts is used to build visual analytics dashboard charts. It consumes live aggregated transaction data, generating responsive pie charts and trend indicators that adapt dynamically when users modify time range selections."

* Axios (v1.16.1)
  - What it is: A promise-based HTTP client for the browser.
  - Why used: Simplifies remote API requests, intercepting authorizations and mapping response payloads.
  - Where used: Centralized API client (`frontend/src/services/api.ts`).
  - Interview Explanation: "Axios serves as our central API driver. We configured custom request interceptors to automatically append JWT bearer tokens from local storage, handling authorization headers globally across all requests."

* FastAPI (v1.0.0 ASGI)
  - What it is: A modern, fast web framework for building APIs with Python.
  - Why used: High-performance routing, automatic OpenAPI schema generation, and asynchronous (async/await) request processing.
  - Where used: Backend entrypoint and routing files (`backend/app/main.py`, `backend/app/routers/*`).
  - Interview Explanation: "FastAPI is the backbone of the backend service. It leverages Python type hints for fast request deserialization and inputs validation, while automatically exposing interactive OpenAPI docs at `/docs`."

* SQLAlchemy (Core & ORM)
  - What it is: A SQL database toolkit and Object-Relational Mapper for Python.
  - Why used: Maps Python database model classes directly to SQLite tables, abstracting raw SQL queries.
  - Where used: Database configuration (`backend/app/database.py`) and model definitions (`backend/app/models/*`).
  - Interview Explanation: "SQLAlchemy is implemented as the Object-Relational Mapper. It establishes Python classes for core business objects like `Transaction` and `Account`, handling joins, constraints, and cascading deletes programmatically."

* SQLite
  - What it is: A self-contained, serverless SQL database engine.
  - Why used: Fast, zero-configuration disk-based relational storage.
  - Where used: Database instances (`backend/money_manager.db`).
  - Interview Explanation: "SQLite was chosen as the relational storage layer. It provides transactional safety, foreign key constraints, and relational queries without the deployment overhead of a standalone server, ideal for desktop and edge environments."

* Pandas & NumPy
  - What it is: Data manipulation and numerical calculation libraries in Python.
  - Why used: Perform lightning-fast matrix manipulations, group-by aggregates, and statistical KPI math on transaction histories.
  - Where used: Analytical calculation services (`backend/app/services/ds_service.py`, `backend/app/services/coach_service.py`).
  - Interview Explanation: "Pandas and NumPy are used in our analytical data service. They ingest list-based database results into structured DataFrames to calculate rolling averages, monthly profit & loss, and mathematical indicators like the Coefficient of Variation for income stability."

* Scikit-Learn
  - What it is: A machine learning library for Python.
  - Why used: Trains predictive models on the user's ledger history to generate forecasts, anomaly flags, and spend clusters.
  - Where used: Machine learning service (`backend/app/services/ds_service.py`).
  - Interview Explanation: "Scikit-Learn drives the machine learning features. We use `LinearRegression` to project future expenses, `IsolationForest` to highlight out-of-pattern anomalies, and `KMeans` to group daily spending aggregates into behavioral clusters (Frugal, Moderate Utility, High-Volume Splurge)."

* Groq, OpenAI, Google Gemini, Anthropic API Clients
  - What it is: Third-party SDK integration endpoints for LLM models.
  - Why used: Connect to powerful models (Llama 3.3, GPT-4o, Gemini 2.5, Claude 3.5) for reasoning over financial data.
  - Where used: LLM service integration (`backend/app/services/llm_service.py`).
  - Interview Explanation: "The application features a multi-model failover AI engine. We wrote custom request handlers that attempt connections to Groq, OpenAI, Gemini, and Anthropic in sequence, selecting the best model based on available API keys and automatically falling back if a service goes offline."

* Jose & Passlib
  - What it is: Cryptographic libraries for JWT signatures and password hashing.
  - Why used: Secure user password verification via bcrypt and session tracking using signed tokens.
  - Where used: Security utilities (`backend/app/services/auth_service.py`, `backend/app/routers/auth.py`).
  - Interview Explanation: "Security is managed using Passlib and python-jose. Passwords are encrypted with bcrypt before database storage, and session authorization is managed using secure JWT tokens containing signed metadata payloads."

4. COMPLETE FEATURE BREAKDOWN
==================================================
* Feature: User Session Authentication
  - Purpose: Secure user access to private ledger details.
  - Frontend Components: `Login.tsx` (Handles registration and login forms), `PinLock.tsx` (Global security PIN block cover).
  - Backend Endpoints:
    * `POST /api/auth/signup` (Creates new user, default categories, and default settings profile)
    * `POST /api/auth/login` (Standard login verifying credentials and returning a JWT token)
    * `GET /api/auth/me` (Returns authenticated user details)
    * `POST /api/auth/verify-pin` (Verifies the security PIN lock)
    * `POST /api/auth/update-pin` (Saves or disables security PIN lock hash)
  - Database Tables: `users`, `settings`, `categories` (seeded defaults).
  - Business Logic: Uses bcrypt to generate password and PIN hashes. Generates signed JWT access tokens containing username and user ID.
  - Interview Explanation: "Session authorization is managed using signed JWT Bearer tokens. When a user logs in, the backend verifies their password hash and returns a token that the frontend stores in local storage and includes in subsequent headers. A global PinLock wrapper intercepts access if the user's settings profile has PIN lock enabled."

* Feature: Multi-Account Balance Ledger
  - Purpose: Manage balances across multiple bank accounts, wallets, and credit cards.
  - Frontend Components: `Accounts.tsx` (Displays accounts list, card types, and transaction histories).
  - Backend Endpoints:
    * `GET /api/accounts` (Retrieves active, non-archived user accounts)
    * `POST /api/accounts` (Registers new account ledger)
    * `PUT /api/accounts/{id}` (Updates account parameters)
    * `DELETE /api/accounts/{id}` (Soft-deletes/archives an account)
  - Database Tables: `accounts`.
  - Business Logic: Accounts are soft-deleted/archived (`is_archived = True`) rather than hard-deleted to preserve historical transaction contexts.
  - Interview Explanation: "The account ledger tracks different account types (Savings, Wallet, CreditCard, Cash). To maintain transactional integrity, deleting an account soft-deletes it by setting `is_archived = True`. This ensures that previous transaction records and KPI charts remain accurate."

* Feature: Transaction Ledger & Balance Reconciliation
  - Purpose: Log incomes, expenses, and internal account transfers with atomic balance updates.
  - Frontend Components: `Transactions.tsx` (Ledger list with filters, addition forms, and transfer forms).
  - Backend Endpoints:
    * `GET /api/transactions` (Retrieves filtered transactions list)
    * `POST /api/transactions` (Logs a transaction and updates account balance)
    * `PUT /api/transactions/{id}` (Modifies transaction details and recalculates account balances)
    * `DELETE /api/transactions/{id}` (Deletes transaction and reverses its balance impact)
    * `POST /api/transactions/bulk-delete` (Deletes multiple transactions and reconciles balances)
    * `POST /api/transactions/transfers` (Logs a transfer between accounts)
  - Database Tables: `transactions`, `transfers`, `accounts`.
  - Business Logic: When an expense transaction is logged, the respective account balance is debited. On updates or deletions, the backend reverses the old transaction impact before applying the new amount. Transfers debit the source account and credit the destination account atomically.
  - Interview Explanation: "Ledger consistency is maintained through programmatic database updates. When a transaction is logged, updated, or deleted, the backend performs calculations to update the respective account balances. On updates, it reverses the previous transaction's impact before applying the new values, preventing balance drift."

* Feature: Budget Category Outlays & Splits
  - Purpose: Manage budgets and allow restructuring categories.
  - Frontend Components: `Categories.tsx` (Displays category limits, current spending, and split interfaces).
  - Backend Endpoints:
    * `GET /api/categories` (Retrieves default and user-specific categories)
    * `POST /api/categories` (Creates a custom category)
    * `PUT /api/categories/{id}` (Updates category name and planned outlay)
    * `POST /api/categories/{id}/split` (Splits a category and migrates transactions)
    * `DELETE /api/categories/{id}` (Deletes a category and re-associates or deletes its transactions)
  - Database Tables: `categories`, `transactions`.
  - Business Logic: Splitting a category creates a new custom category and updates related transactions' foreign keys based on the user's preference (e.g., manual mapping or bulk migration). Deleting a category requires migrating referencing transactions to a destination category or deleting them.
  - Interview Explanation: "The budget category system supports dynamic restructuring. A category split updates the parent category's name and planned outlay, creates a new sub-category, and migrates related transaction references. This ensures database references remain valid during category reorganization."

* Feature: Data Science & ML Engine
  - Purpose: Calculate statistical KPIs and run predictive models.
  - Frontend Components: `Analytics.tsx` (Displays KPI cards, pie charts, forecasts, outlier tables, and cluster cards).
  - Backend Endpoints:
    * `GET /api/analytics/kpis` (Returns basic metrics like mean spend, median spend, income stability, and savings rates)
    * `GET /api/analytics/ml` (Returns expense forecasts, outlier anomalies, and behavioral clusters)
  - Database Tables: `transactions`, `accounts`.
  - Business Logic: Uses Scikit-Learn to run predictions. Projects next month's bills using `LinearRegression`, identifies outlier transactions using `IsolationForest` and Z-scores, and clusters daily spending aggregates using `KMeans`.
  - Interview Explanation: "The analytics engine integrates data science models directly with the transactional database. It runs a Linear Regression model on aggregated monthly expenses to forecast future spend, uses Isolation Forest and Z-scores to flag anomaly outliers, and groups daily aggregates using KMeans clustering into behavioral categories."

* Feature: AI Advisor Coach
  - Purpose: Provide interactive, data-grounded personal finance guidance.
  - Frontend Components: `CoachWidget.tsx` (Conversational window with tabs for Chat, Diagnostics, Planner, and Monthly Reviews).
  - Backend Endpoints:
    * `GET /api/coach/insights` (Generates review grades, savings suggestions, and warning checklists)
    * `POST /api/coach/chat` (Processes chat messages and returns AI advisor answers)
  - Database Tables: `transactions`, `accounts`, `categories`, `settings`.
  - Business Logic: The backend builds a structured JSON context summarizing the user's financial details. The request classifies query intent to select the appropriate agent prompt (e.g., CFO, Budget, Health) and queries the LLM provider, falling back to a deterministic template if offline.
  - Interview Explanation: "The AI Coach runs a multi-agent system grounded in real-time user database context. When a user sends a query, the backend classifies its intent to route it to a specialized agent persona. The LLM receives a prompt containing the user's actual financial data, ensuring responses contain accurate numbers."

5. DATABASE DESIGN
==================================================
* Table: users
  - Table Name: `users`
  - Columns:
    * `id` (Integer): Primary Key (Auto-Increment, indexed)
    * `username` (String): Unique, indexed, not null
    * `email` (String): Unique, indexed, not null
    * `hashed_password` (String): Not null
    * `created_at` (DateTime): Default UTC now
  - Primary Key: `id`
  - Foreign Keys: None
  - Relationships:
    * `accounts` (One-to-Many with `Account` via `user_id`, cascading delete)
    * `categories` (One-to-Many with `Category` via `user_id`, cascading delete)
    * `transactions` (One-to-Many with `Transaction` via `user_id`, cascading delete)
    * `transfers` (One-to-Many with `Transfer` via `user_id`, cascading delete)
    * `notifications` (One-to-Many with `Notification` via `user_id`, cascading delete)
    * `settings` (One-to-One with `Settings` via `user_id`, cascading delete)
    * `tickets` (One-to-Many with `SupportTicket` via `user_id`, cascading delete)
    * `snapshots` (One-to-Many with `AnalyticsSnapshot` via `user_id`, cascading delete)
  - Why it exists: Core user registration records and password hashes.

* Table: settings
  - Table Name: `settings`
  - Columns:
    * `id` (Integer): Primary Key (Auto-Increment, indexed)
    * `user_id` (Integer): Unique, Foreign Key to `users.id` (ondelete="CASCADE"), not null
    * `pin_hash` (String): Nullable (stores hashed 4-digit PIN)
    * `is_pin_enabled` (Boolean): Default False
    * `auto_lock_duration` (Integer): Default 300 seconds
    * `session_timeout` (Integer): Default 3600 seconds
    * `currency` (String): Default "INR"
    * `language` (String): Default "English"
    * `date_format` (String): Default "DD-MM-YYYY"
    * `notification_pref` (String): Default "Email"
    * `theme` (String): Default "FinancePro"
    * `theme_customization` (String): Default "{}" (JSON string)
    * `hide_balance` (Boolean): Default False
    * `lock_analytics` (Boolean): Default False
    * `created_at` (DateTime): Default UTC now
  - Primary Key: `id`
  - Foreign Keys: `user_id` references `users.id`
  - Relationships: `user` (Many-to-One back to `User`)
  - Why it exists: Custom user configurations, visual preferences, and PIN hashes.

* Table: accounts
  - Table Name: `accounts`
  - Columns:
    * `id` (Integer): Primary Key (Auto-Increment, indexed)
    * `user_id` (Integer): Foreign Key to `users.id` (ondelete="CASCADE"), not null
    * `name` (String): Indexed, not null (e.g. "Main Savings")
    * `bank_name` (String): Not null (e.g. "HDFC")
    * `type` (String): Not null ("Savings", "Current", "CreditCard", "Cash", "Wallet")
    * `balance` (Float): Default 0.0
    * `currency` (String): Default "INR"
    * `is_archived` (Boolean): Default False
    * `created_at` (DateTime): Default UTC now
  - Primary Key: `id`
  - Foreign Keys: `user_id` references `users.id`
  - Relationships:
    * `user` (Many-to-One back to `User`)
    * `transactions` (One-to-Many with `Transaction` via `account_id`, cascading delete)
    * `transfers_sent` (One-to-Many with `Transfer` via `source_account_id`, cascading delete)
    * `transfers_received` (One-to-Many with `Transfer` via `destination_account_id`, cascading delete)
  - Why it exists: Tracks account types, bank details, and active balances.

* Table: categories
  - Table Name: `categories`
  - Columns:
    * `id` (Integer): Primary Key (Auto-Increment, indexed)
    * `user_id` (Integer): Foreign Key to `users.id` (ondelete="CASCADE"), nullable (NULL indicates system defaults)
    * `name` (String): Not null
    * `type` (String): Not null ("Expense", "Income")
    * `planned_outlay` (Float): Default 0.0 (planned monthly budget limit)
    * `is_custom` (Boolean): Default False
    * `created_at` (DateTime): Default UTC now
  - Primary Key: `id`
  - Foreign Keys: `user_id` references `users.id` (nullable)
  - Relationships:
    * `user` (Many-to-One back to `User`)
    * `transactions` (One-to-Many with `Transaction` via `category_id`)
  - Why it exists: Classifies user cash flow and defines monthly budget limits.

* Table: transactions
  - Table Name: `transactions`
  - Columns:
    * `id` (Integer): Primary Key (Auto-Increment, indexed)
    * `user_id` (Integer): Foreign Key to `users.id` (ondelete="CASCADE"), not null
    * `account_id` (Integer): Foreign Key to `accounts.id` (ondelete="CASCADE"), not null
    * `category_id` (Integer): Foreign Key to `categories.id` (ondelete="RESTRICT"), not null
    * `type` (String): Not null ("Expense", "Income")
    * `amount` (Float): Not null
    * `description` (String): Nullable
    * `date` (DateTime): Not null, default UTC now
    * `created_at` (DateTime): Default UTC now
  - Primary Key: `id`
  - Foreign Keys:
    * `user_id` references `users.id`
    * `account_id` references `accounts.id`
    * `category_id` references `categories.id`
  - Relationships:
    * `user` (Many-to-One back to `User`)
    * `account` (Many-to-One back to `Account`)
    * `category` (Many-to-One back to `Category`)
  - Why it exists: Historical record of transaction outlays and receipts.

* Table: transfers
  - Table Name: `transfers`
  - Columns:
    * `id` (Integer): Primary Key (Auto-Increment, indexed)
    * `user_id` (Integer): Foreign Key to `users.id` (ondelete="CASCADE"), not null
    * `source_account_id` (Integer): Foreign Key to `accounts.id` (ondelete="CASCADE"), not null
    * `destination_account_id` (Integer): Foreign Key to `accounts.id` (ondelete="CASCADE"), not null
    * `amount` (Float): Not null
    * `description` (String): Nullable
    * `date` (DateTime): Not null, default UTC now
    * `created_at` (DateTime): Default UTC now
  - Primary Key: `id`
  - Foreign Keys:
    * `user_id` references `users.id`
    * `source_account_id` references `accounts.id`
    * `destination_account_id` references `accounts.id`
  - Relationships:
    * `user` (Many-to-One back to `User`)
    * `source_account` (Many-to-One with `Account` mapping to source account)
    * `destination_account` (Many-to-One with `Account` mapping to target account)
  - Why it exists: Tracks fund transfers between a user's accounts.

* Table: support_tickets
  - Table Name: `support_tickets`
  - Columns:
    * `id` (Integer): Primary Key (Auto-Increment, indexed)
    * `user_id` (Integer): Foreign Key to `users.id` (ondelete="CASCADE"), not null
    * `subject` (String): Not null

7. AI ASSISTANT ARCHITECTURE
==================================================
The AI assistant runs a multi-agent system grounded in real-time user database context.

Detailed Request-Response Lifecycle:
  1. Frontend Post: The frontend sends a user message and conversation history to `/api/coach/chat`.
  2. Context Retrieval: The backend instantiates `CoachService(db, user.id)` to fetch the user's accounts, categories, and transactions.
  3. Context Assembly: `FinancialContextBuilder` aggregates this data into a structured JSON payload containing monthly summaries, budget analysis, category trends, health sub-scores, and recent transactions.
  4. Context Trimming: If the context size exceeds 6,000 characters, it is truncated to avoid context window overflows.
  5. Intent Classification: The backend classifies the query's intent (e.g., CFO review, budget advice, affordability check) using keyword analysis.
  6. Agent Selection: The classified intent is mapped to a specialized prompt agent:
     - `cfo`: Monthly financial audits with tabular reports.
     - `health`: Health score diagnostic diagnostics.
     - `savings`: Discretionary savings optimization plans.
     - `budget`: Outlay assessments and category reductions.
     - `affordability`: Cost evaluations against net worth.
     - `master`: General financial guidelines and scenarios.
  7. Prompt Building: The system merges the base instructions with the selected agent prompt and context JSON.
  8. Multi-Model Routing & Failover: The backend calls the best available LLM provider based on configured API keys:
     - Groq API (`llama-3.3-70b-versatile`)
     - OpenAI API (`gpt-4o`)
     - Google Gemini API (`gemini-2.5-flash`)
     - Anthropic API (`claude-3-5-sonnet-20241022`)
     - Local Fallback: Renders a deterministic text template if all remote API calls fail.
  9. Response Sanitization: The reply is sanitized to convert markdown titles to HTML tags, strip emojis, and remove duplicate sentences.
  10. Rendering: The backend returns raw HTML (`{"reply": reply}`), which the frontend renders directly in the chat window.

Step-by-Step AI Execution Flow Chart:
   [User Inputs Query]
          │
          ▼
   [FastAPI Route: /coach/chat]
          │
          ▼
   [Retrieve DB Data (CoachService)]
          │
          ▼
   [Assemble Context (FinancialContextBuilder)]
          │
          ▼
   [Context Trimmer (< 6,000 Chars)]
          │
          ▼
   [Intent Classifier (e.g., 'CFO Audit')] ──> [Select Agent Prompt (e.g., 'cfo')]
          │
          ▼
   [Query LLM Chain (Groq ──> OpenAI ──> Gemini ──> Anthropic ──> Local Template)]
          │
          ▼
   [Sanitize Output (Remove Markdown, Emojis, and Duplicates)]
          │
          ▼
   [Return HTML Reply] ──> [React Renders HTML]

8. END-TO-END APPLICATION FLOW
==================================================
* Flow A: User Login
  - Step 1 (Frontend): User enters credentials on the `/login` screen. Axios calls `POST /api/auth/login`.
  - Step 2 (Backend): FastAPI retrieves user records by username or email. It verifies the password using bcrypt.
  - Step 3 (Backend): If verified, it generates a JWT containing user ID metadata and returns it to the client.
  - Step 4 (Frontend): Zustand saves the token to local storage and updates `isAuthenticated = true`.
  - Step 5 (Frontend): The app redirects to `/`. The API client automatically includes the token in subsequent headers.

* Flow B: Account Creation
  - Step 1 (Frontend): User enters account details (e.g. Savings, Bank: HDFC, Balance: 50,000) on the `/accounts` page and clicks Save.
  - Step 2 (Frontend): Axios sends a `POST` request containing the parameters to `/api/accounts`.
  - Step 3 (Backend): FastAPI verifies the user's JWT token, instantiates an `Account` model, and saves it to the SQLite database.
  - Step 4 (Backend): The endpoint returns the created record.
  - Step 5 (Frontend): React Query invalidates the `accounts` cache, triggering a refresh of the page and net worth metrics.

* Flow C: Transaction Creation & Reconciliation
  - Step 1 (Frontend): User opens the Transaction form, inputs details (Amount: 1,500, Type: Expense, Category: Food, Account: HDFC), and clicks Save.
  - Step 2 (Frontend): Axios sends a `POST` request containing the parameters to `/api/transactions`.
  - Step 3 (Backend): FastAPI verifies the token, checks that the account and category exist, and logs a new `Transaction` record.
  - Step 4 (Backend): The backend adjusts the associated account balance based on the transaction type:
    - Expense: `account.balance -= amount`
    - Income: `account.balance += amount`
  - Step 5 (Backend): FastAPI commits the database transaction and returns the transaction record.
  - Step 6 (Frontend): React Query invalidates the `transactions` and `accounts` caches, refreshing ledger tables and KPI cards.

* Flow D: Budget Tracking & Warnings
  - Step 1 (Backend): When logging a transaction, the backend checks the category's planned monthly outlay.
  - Step 2 (Backend): It aggregates the current month's expenses for that category. If spending exceeds the budget, it creates a database notification.
  - Step 3 (Frontend): When the dashboard loads, React Query fetches active alerts via `GET /api/support/notifications`.
  - Step 4 (Frontend): Alerts are displayed in the UI (e.g., "Warning: Food is at 95% of its monthly budget").

* Flow E: Data Science & ML Predictions
  - Step 1 (Frontend): User navigates to `/analytics`.
  - Step 2 (Frontend): React Query sends requests to `/api/analytics/kpis` and `/api/analytics/ml`.
  - Step 3 (Backend): FastAPI retrieves the user's transactions and accounts database records.
  - Step 4 (Backend): In `ds_service.py`, data is parsed into a Pandas DataFrame:
    - Standard KPIs (average spend, median spend, CV) are calculated.
    - An Isolation Forest model identifies anomalous spending outliers.
    - KMeans clusters daily aggregates into Frugal/Splurge groups.
    - Linear Regression projects future expenses.
  - Step 5 (Backend): Returns the statistical and model outputs to the client.
  - Step 6 (Frontend): Renders the results in pie charts, trend lines, and KPI cards.

* Flow F: AI Assistant conversational queries
  - Step 1 (Frontend): User opens the Coach panel and enters a query (e.g. "Am I overspending?").
  - Step 2 (Frontend): The UI displays a typing indicator and sends a `POST` request to `/api/coach/chat`.
  - Step 3 (Backend): The router builds a JSON context of the user's finances and classifies the query's intent (e.g. `spending_risks` mapping to `health`).
  - Step 4 (Backend): It queries the LLM provider, providing the financial context and chat history.
  - Step 5 (Backend): The LLM returns HTML-formatted advice grounded in the user's actual database numbers.
  - Step 6 (Backend): The response is sanitized and returned to the client.
  - Step 7 (Frontend): The typing indicator is removed, and the raw HTML is rendered in the chat window.

10. CHALLENGES & SOLUTIONS
==================================================
* Challenge 1: LLM Context Window Bloat and Token Limit Failures
  - Problem: Users with extensive transaction histories generated JSON contexts that exceeded LLM token limits, causing API errors.
  - Root Cause: Serializing all raw database records exceeded context window thresholds.
  - Solution: Implemented a progressive truncation strategy in `FinancialContextBuilder` that trims verbose arrays (slicing transactions to 10, removing account details, dropping recurring tags) if the context string exceeds 6,000 characters.
  - Learning: Always implement context-pruning algorithms on dynamic datasets before passing them to LLMs to prevent token limit errors.

* Challenge 2: Transaction Balance Drift During Updates
  - Problem: Modifying transaction amounts or accounts resulted in incorrect account balances.
  - Root Cause: Directly saving updates failed to reconcile the database balances with previous transaction amounts.
  - Solution: Wrote reconciliation logic in `transactions.py` that reverses the old transaction amount from the original account, commits, and then applies the new transaction amount to the target account.
  - Learning: Changing records in double-entry ledgers requires reversing the old event impact before executing the new one.

* Challenge 3: Orphaned Transactions on Category Deletions
  - Problem: Deleting a category with active transactions caused foreign key constraint violations in SQLite.
  - Root Cause: SQLite block constraints prevented deleting parent records that were referenced by active transaction records.
  - Solution: Developed a category dependencies endpoint and updated `categories.py` to allow users to either delete related transactions or migrate them to a new category.
  - Learning: Relational integrity checks require implementing migration options for child records before deleting parent keys.

* Challenge 4: Handling LLM API Rate Limits and Outages
  - Problem: Rate limits on external LLM services caused errors in the conversational advisor.
  - Root Cause: Relying on a single third-party API created a single point of failure.
  - Solution: Implemented a failover chain in `llm_service.py` that queries alternative providers (Groq -> OpenAI -> Gemini -> Anthropic) and falls back to a deterministic rule-based template if offline.
  - Learning: High-availability AI services require robust fallback strategies to ensure application availability during API outages.

  - `POST /api/transactions`: Logs transaction, reconciles balances.
  - `POST /api/categories/{id}/split`: Splits categories, migrates data.
  - `GET /api/analytics/ml`: Returns models forecasts and anomalies.
  - `POST /api/coach/chat`: Processes grounded conversational queries.

* Critical Design Decisions:
  - Context Trimming: Reduces payload sizes below 6,000 characters to prevent token errors.
  - Multi-Model Failover: Queries Groq -> OpenAI -> Gemini -> Anthropic -> local fallback templates.
  - Relational Integrity: Uses CASCADE delete on users, and RESTRICT on categories to protect transaction data.
  - Soft Delete: Archives accounts (`is_archived = True`) to preserve transaction histories.
  - Raw HTML Responses: AI replies in raw HTML to bypass markdown rendering overhead.
